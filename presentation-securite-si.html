Fondamentaux SSI

Transformation numérique · Enjeux · CIA · Coût qualité

Pourquoi la sécurité est au cœur des projets numériques modernes

Contexte & Enjeux

Le logiciel est partout

  • Du bureau aux objets connectés (IoT)
  • Santé, Automobile, Finance, Industrie
  • Systèmes critiques à fort impact humain
  • Visibilité totale par les utilisateurs/clients
  • Systèmes de plus en plus complexes & interconnectés

Nouveaux paradigmes

  • Cloud natif · Microservices · API-first
  • DevOps — livraisons continues (CI/CD)
  • Chaînes de dépendances tierces étendues
  • Équipes distribuées à l'échelle mondiale
82% Des organisations ont subi une faille liée au code applicatif (CISA 2023)
4.5M$ Coût moyen d'une violation de données en 2023 (IBM)
30× Coût d'un correctif en production vs en conception
95% Des failles sont dues à des erreurs de code ou configuration (Gartner)

Définitions Clés — La Triade CIA

C Confidentialité Accès réservé aux personnes autorisées
I Intégrité Données non altérées sans autorisation
A Disponibilité Système accessible quand nécessaire
Menace — Cause potentielle d'un incident (attaquant, erreur humaine, catastrophe, malware…)
Vulnérabilité — Faiblesse exploitable dans un système, une application ou un processus
Risque = Menace × Vulnérabilité × Impact → probabilité × gravité
Contrôle — Mesure technique ou organisationnelle réduisant un risque identifié
Système d'Information (SI) : ensemble organisé de ressources (matériels, logiciels, données, procédures, personnel) permettant de collecter, stocker, traiter et diffuser l'information.

Pourquoi la sécurité dans le développement est cruciale ?

"Dans le logiciel, la plupart des défauts proviennent du développeur. Si nous pouvons arrêter d'introduire des défauts au fur et à mesure, nous obtenons un logiciel bien meilleur." — Capers Jones

Conséquences Business

  • Amendes réglementaires (RGPD, PCI-DSS)
  • Perte de clientèle & churn
  • Arrêt d'activité (downtime)
  • Atteinte à la réputation durable
  • Sanctions pénales dirigeants

Conséquences Techniques

  • Fuite de données personnelles (PII)
  • Corruption / chiffrement données (ransomware)
  • Déni de service (DoS/DDoS)
  • Prise de contrôle système (RCE)
  • Supply chain compromise

Bénéfices d'investir tôt

  • Réduction drastique des coûts
  • Conformité intégrée & auditable
  • Confiance des utilisateurs
  • Time-to-market accéléré
  • Avantage concurrentiel

SSDLC · Security by Design · Threat Modeling · Livrables

SDLC vs SSDLC — Changement de Paradigme

SDLC Traditionnel

  • Sécurité ajoutée en fin de cycle
  • Tests de pénétration one-shot (avant release)
  • Équipe sécurité isolée des devs
  • Correctifs coûteux en production
  • Approche : "construire, échouer, réparer"
  • Compliance comme formalité

SSDLC — Secure SDLC

  • Sécurité intégrée à chaque phase
  • Threat modeling dès la conception
  • Collaboration DevSecOps continue
  • Correction précoce et moins coûteuse
  • Approche : "concevoir, construire, livrer"
  • Compliance by design
Paradigme SSDLC : Intégrer la sécurité dès la naissance du projet — à chaque étape, jusqu'à la fin du cycle de vie.

Les Phases du SSDLC

📋

1. Exigences (Requirements)

Exigences de sécurité fonctionnelles & non-fonctionnelles · Use cases d'abus (abuse cases) · Classification des données · Conformité réglementaire (RGPD, PCI-DSS, ISO 27001, SOC2) · Exigences financières & de conformité

🎨

2. Conception (Design)

Threat Modeling (STRIDE, PASTA, LINDDUN) · Architecture sécurité (fonctionnelle, applicative, technique) · Principes Security by Design · Diagrammes de flux de données (DFD) · Revue d'architecture sécurisée

💻

3. Implémentation (Coding)

Standards de codage sécurisé (CERT, MISRA, OWASP ASVS) · Revue de code sécurité (peer review + SAST) · Gestion des secrets (Vault, env vars) · Dépendances sécurisées (SCA) · Validation des entrées

🧪

4. Tests (Testing)

Analyse dynamique (DAST) · Tests de pénétration · Tests de régression sécurité · Fuzzing · Tests de charge & résilience · Scans de vulnérabilités

🚀

5. Déploiement & Maintenance

Hardening de configuration · Gestion des patches · Surveillance continue (SIEM, IDS) · Plan de réponse aux incidents · Revue post-incident · Rotation des secrets

Security by Design — Les 9 Principes Fondamentaux

① Moindre Privilège

Chaque composant n'accède qu'aux ressources strictement nécessaires à sa fonction. Ni plus, ni moins.

② Défense en Profondeur

Plusieurs couches de contrôles indépendantes. Si une couche échoue, les autres protègent encore.

③ Sécurité par Défaut

La configuration par défaut est la plus sécurisée. L'utilisateur active des fonctions, pas la sécurité.

④ Fail Securely

En cas d'erreur, le système revient à un état sécurisé — refus d'accès par défaut, pas d'exposition.

⑤ Ne jamais faire confiance aux entrées

Toute entrée externe (user, API, BDD, fichier) est potentiellement malveillante — valider & assainir.

⑥ Open Design

La sécurité ne repose pas sur le secret de la conception. Pas de security through obscurity.

⑦ Réutilisation des Contrôles

Utiliser des librairies et frameworks sécurité éprouvés plutôt que de réinventer la roue.

⑧ Séparation des Privilèges

Plusieurs conditions doivent être réunies pour accéder à une ressource critique (double contrôle).

⑨ Logging & Auditabilité

Journaliser les événements de sécurité significatifs pour la détection, la forensique et la conformité.

① Moindre Privilège — Cas Pratique

Contexte : PayBuddy, une fintech, configure son application web avec un seul utilisateur BD qui a tous les droits.

Problème — Trop de droits

L'application utilise l'utilisateur db_admin pour toutes les opérations — lecture ET écriture ET suppression.

-- Utilisateur BD avec tous les droits
GRANT SELECT, INSERT, UPDATE,
      DELETE, DROP, CREATE
  ON payments.* TO 'app_user';

Attaque — SQL Injection

Un attaquant injecte une commande SQL via un champ de formulaire :

-- Payload injecté dans un formulaire
'; DROP TABLE payments; --

-- Résultat : toute la table supprimée !
-- L'app avait le droit DROP... donc ça marche.

→ Perte totale des données de paiement

Solution — Principe du moindre privilège

Créer des rôles distincts selon le besoin :

-- Lecture seule pour l'API publique
GRANT SELECT ON payments.* TO 'api_user';
-- Écriture uniquement pour le service de paiement
GRANT INSERT, UPDATE ON payments.*
  TO 'payment_svc';
  • Chaque service a son propre compte BD
  • Aucun service n'a les droits DROP ou CREATE
  • Si une injection SQL réussit → aucun dommage structurel
  • Appliquer le même principe aux APIs, fichiers, secrets

② Protection des Données — Cas Pratique

Contexte : PayBuddy expose une API de paiement. Les numéros de carte transitent en clair entre le client et le serveur.

Problème — Données en clair

L'API tourne en HTTP (pas HTTPS). La réponse retourne le numéro complet :

// Réponse API — DANGEREUX
{
  "user": "John Doe",
  "card": "4532 1234 5678 9420",
  "cvv": "123",
  "status": "confirmed"
}

Attaque — Man-in-the-Middle

Sur un réseau public (café, aéroport), un attaquant intercepte le trafic :

  • Wireshark capture les paquets HTTP
  • Numéro de carte lu en clair
  • CVV disponible → fraude immédiate
  • Violation PCI-DSS → amendes lourdes

Solution — TLS + Masquage des données

Transport : HTTPS obligatoire (TLS 1.3)

// Réponse API — SÉCURISÉ
{
  "user": "John Doe",
  "card": "XXXXXXXXXXXX9420",
  "status": "confirmed"
}
  • TLS chiffre le transport → interception inutile
  • Masquer les données sensibles dans les réponses
  • Ne jamais stocker les CVV (interdit PCI-DSS)
  • Chiffrer les numéros de carte au repos (AES-256)
  • Tokeniser avec des vault de paiement (Stripe, Braintree)

② Défense en Profondeur — Cas Pratique

Principe : Plusieurs couches de sécurité indépendantes — si une couche tombe, les autres résistent encore.
7
Périmètre réseau (Border) — Firewall périmétrique, IDS/IPS, filtrage BGP
6
Accès physique — Badge RFID, caméras, salle serveurs verrouillée
5
Firewall applicatif (WAF) — Bloque SQLi, XSS, CSRF au niveau HTTP
4
Segmentation réseau — VLAN, zones DMZ, microsegmentation
3
Hôtes patchés — OS/libs à jour, hardening CIS, antivirus EDR
2
Validation des entrées — Validation, sanitisation, paramètres préparés
1
BD chiffrée — Chiffrement au repos (AES-256), accès minimal, audit trail
Règle d'or : Un attaquant qui franchit la couche 7 rencontre encore 6 obstacles. Chaque couche ralentit, détecte et limite l'impact.

⑨ Logging — Les 2 Pièges Opposés

Paradoxe : Trop peu de logs = attaque invisible. Trop de logs (avec données sensibles) = violation RGPD.

Piège 1 — Logs vides

Un attaquant tente 10 000 mots de passe sur un compte (brute force). Les logs ne tracent rien :

# Fichier auth.log — VIDE
# Aucune trace des 10 000 tentatives
# L'attaque n'est jamais détectée
# Le compte est compromis après 3h

→ Intrusion non détectée pendant des semaines

Piège 2 — Trop d'infos dans les logs

L'application logue tout, y compris les données sensibles :

# auth.log — DANGEREUX
[INFO] Login attempt: user=john
  password=MyP@ssw0rd123
  card=4532123456789420
  ip=192.168.1.10

→ Violation RGPD · Exposition si log volé

Solution — Logs utiles sans données sensibles

# auth.log — CORRECT
[WARN] Failed login: user=john
  attempt=3/5 ip=192.168.1.10
  timestamp=2024-01-15T14:23:01Z
[ALERT] Account locked: user=john
  reason=max_attempts_exceeded
  • Logguer les événements, jamais les secrets
  • Détecter le brute force (seuil d'alertes)
  • Jamais : mots de passe, tokens, cartes, clés
  • Horodater et centraliser (SIEM)

③ Sécurité par Défaut — Cas Pratique

Contexte : Une application de gestion d'équipe est livrée avec une politique de mots de passe trop permissive par défaut.

Configuration par défaut — Dangereuse

Paramètres de sécurité du mot de passe dans config.yml :

password_policy:
  min_length: 6          # Trop court !
  require_uppercase: 0   # Non requis
  require_numbers: 0     # Non requis
  require_symbols: 0     # Non requis
  max_attempts: 0        # Illimité !

→ Mots de passe comme abc123 acceptés

Attaque — Brute Force réussie

  • Dictionnaire des 1 000 mots de passe les plus communs
  • Aucune limite de tentatives → script automatisé
  • Mot de passe password trouvé en 3 secondes
  • Accès admin obtenu → données de toute l'équipe exposées
  • L'admin n'a jamais changé le mot de passe par défaut

Solution — Politique sécurisée par défaut

password_policy:
  min_length: 12         # ≥ 12 caractères
  require_uppercase: 1   # Obligatoire
  require_numbers: 1     # Obligatoire
  require_symbols: 1     # Obligatoire
  max_attempts: 5        # Verrouillage
  lockout_duration: 15m  # 15 min
  • La config la plus sécurisée est la config par défaut
  • L'admin doit réduire la sécurité intentionnellement
  • Forcer le changement du mot de passe initial
  • Activer MFA par défaut pour les comptes admin

Gestion Robuste des Erreurs — Cas Pratique

Contexte : Une API expose une stack trace complète lors d'une erreur — un cadeau pour un attaquant.

Erreur avec Stack Trace exposée

Requête : GET /api/user?id=abc

HTTP 500 Internal Server Error
java.sql.SQLException: Invalid column type
  at org.hibernate.type.IntegerType.get
     (IntegerType.java:44)
  at com.paybuddy.dao.UserDAO.findById
     (UserDAO.java:87)
  Connection: jdbc:mysql://db-prod:3306/users
  Framework: Spring Boot 2.7.1 (Tomcat 9.0)

→ Framework, version, host BD, structure du code… tout est révélé !

Ce que l'attaquant apprend

  • Spring Boot 2.7.1 → CVE connues à cibler
  • Hibernate ORM → vecteur d'injection HQL
  • MySQL sur db-prod:3306 → topologie réseau
  • UserDAO.java:87 → logique métier exposée
  • Cartographie complète de la stack technique en 1 requête

Solution — Erreurs génériques + logs internes

// Réponse API — SÉCURISÉ
HTTP 500
{
  "error": "Une erreur est survenue.",
  "code": "ERR_500",
  "ref": "ERR-2024-001547"
}
// Logs INTERNES (jamais exposés)
ERROR [UserDAO] id=abc invalid type
  trace: ...stack complète ici...
  • Réponse publique : message générique + code de référence
  • Stack trace uniquement dans les logs internes
  • Désactiver le debug mode en production
  • Gérer toutes les exceptions (ne pas laisser propager)

⑥ Open Design — Éviter la Sécurité par l'Obscurité

Idée reçue : "Si je change le port SSH de 22 à 6666, personne ne le trouvera."

Fausse sécurité — Port inhabituel

Configuration du serveur SSH :

# /etc/ssh/sshd_config
Port 6666                    # "Port secret"
PasswordAuthentication yes   # Mot de passe autorisé
# Mot de passe root : "dragon"

Le responsable IT pense que personne ne cherchera sur ce port.

Attaque — Nmap révèle tout

# Scan complet de tous les ports
$ nmap -p 1-65535 192.168.1.10

PORT     STATE  SERVICE
22/tcp   closed ssh
6666/tcp open   ssh   # Trouvé en 30 sec !
# OpenSSH 8.2 → CVE connues
# Brute force avec "dragon" → accès root

→ Nmap scan tous les 65 535 ports en moins d'une minute

Solution — Sécurité réelle, pas par l'obscurité

# /etc/ssh/sshd_config — SÉCURISÉ
Port 22                         # Port standard OK
PasswordAuthentication no       # Clés SSH uniquement
PermitRootLogin no              # Jamais root direct
AllowUsers deploy@10.0.0.0/24  # Whitelist IP
MaxAuthTries 3
  • Désactiver l'auth par mot de passe → clés SSH uniquement
  • Principe d'Open Design : la sécurité résiste même si la config est connue
  • Ajouter un fail2ban, VPN, ou bastion host
  • Le port inhabituel n'est pas une défense — c'est de l'obscurantisme

⑦ Réutilisation des Contrôles — Session Prévisible

Contexte : Une équipe développe son propre mécanisme de session au lieu d'utiliser un framework éprouvé.

Session ID maison — Prévisible

Le développeur génère un identifiant de session "unique" :

// SessionGenerator.java — DANGEREUX
String sessionId = user.getInitials()
  + dateFormat("ddMMyyyy");

// John Doe se connecte le 23/11/2018
// sessionId = "JD23112018"

→ Identifiant totalement prévisible !

Attaque — Session Hijacking

L'attaquant connaît le nom de la victime et la date de connexion :

# L'attaquant sait que John Doe
# s'est connecté aujourd'hui :
# sessionId = "JD" + today() = "JD23112018"

curl -H "Cookie: sessionId=JD23112018" \
     https://app.example.com/dashboard
# → Accès au compte de John Doe !

Solution — Utiliser les contrôles éprouvés

// SÉCURISÉ — UUID cryptographiquement sûr
// Utiliser le framework (Spring Security,
// Express-session, Django sessions…)
String sessionId = UUID.randomUUID()
  .toString(); // "a3f9e2b1-7c4d-..."

// Ou mieux : JWT signé
String token = Jwts.builder()
  .setSubject(user.getId())
  .setExpiration(new Date(now + 3600000))
  .signWith(secretKey).compact();
  • Ne jamais implémenter ses propres mécanismes d'auth/session
  • Utiliser des générateurs cryptographiquement sûrs (SecureRandom)
  • Session ID = au moins 128 bits d'entropie
  • Renouveler le token après connexion (Session Fixation)

④ Fail Securely — Cas Pratique

Principe : Quand quelque chose échoue, le système doit se replier sur un état sécurisé — jamais sur un état ouvert par défaut.

Fail Open — Dangereux

Le service d'authentification est temporairement indisponible :

// DANGEREUX — Fail Open
boolean isAuthorized = false;
try {
  isAuthorized = authService.check(user);
} catch (ServiceUnavailableException e) {
  // Auth service down ? On laisse passer !
  isAuthorized = true; // ← CATASTROPHIQUE
  log.warn("Auth service down, allowing...");
}
if (isAuthorized) grantAccess();

→ Panne = accès libre à tous !

Fail Secure — Refus par défaut

// SÉCURISÉ — Fail Secure
boolean isAuthorized = false; // défaut = refus
try {
  isAuthorized = authService.check(user);
} catch (ServiceUnavailableException e) {
  // Auth service down ? On refuse l'accès.
  log.error("Auth service unavailable");
  throw new AccessDeniedException(
    "Service temporairement indisponible");
}
if (isAuthorized) grantAccess();

️ Autres exemples Fail Secure

Firewall en panne
→ Bloquer tout le trafic, pas l'autoriser
Service d'autorisation down
→ Refuser l'accès, pas l'accorder
Validation échoue
→ Rejeter la donnée, jamais la laisser passer

⑤ Ne Jamais Faire Confiance aux Entrées — Injection SQL

Règle fondamentale : Toute entrée venant de l'extérieur (utilisateur, API, base de données, fichier) est potentiellement malveillante.

Concaténation directe — Injection SQL

// DANGEREUX — Concaténation
String query = "SELECT * FROM users "
  + "WHERE username = '" + username + "'";

// Entrée attaquant :
// username = "admin' OR '1'='1"

// Requête générée :
// SELECT * FROM users
// WHERE username = 'admin' OR '1'='1'
// → Retourne TOUS les utilisateurs !

Requête préparée — Paramétrisée

// SÉCURISÉ — PreparedStatement
PreparedStatement ps = conn.prepareStatement(
  "SELECT * FROM users WHERE username = ?");
ps.setString(1, username);
// Le driver JDBC échappe automatiquement
// Même si username = "admin' OR '1'='1"
// → Cherche littéralement ce nom d'utilisateur

// Spring Data JPA (encore mieux)
@Query("SELECT u FROM User u
        WHERE u.username = :name")
User findByUsername(@Param("name") String name);

Règles de validation des entrées

Whitelist
Définir ce qui est autorisé, pas ce qui est interdit
Sanitiser
Encoder la sortie selon le contexte (HTML, SQL, Shell)
Jamais confiance
Même les données issues de la BD doivent être validées

Échappement & Assainissement — Principes

Distinction clé : Sanitiser = nettoyer/supprimer les données dangereuses en entrée. Échapper = neutraliser les caractères spéciaux en sortie, selon le contexte.

🧹 Assainissement (Sanitisation)

Nettoyer les données avant de les traiter : supprimer ou remplacer les caractères indésirables.

// Supprimer les balises HTML d'une entrée utilisateur
// Bibliothèque : OWASP Java HTML Sanitizer
String safe = HtmlPolicyBuilder
  .allowElements("b", "i", "p")  // whitelist
  .allowUrlProtocols("https")
  .toFactory()
  .sanitize(userInput);

// Input  : <script>alert('xss')</script>Bonjour
// Output : Bonjour  (balise <script> supprimée)
  • Supprimer les balises <script>, onclick, onerror…
  • Utiliser une whitelist (ce qui est permis), pas une blacklist
  • Valider le format : email, téléphone, URL, nombre…

🔐 Échappement (Encoding)

Transformer les caractères spéciaux avant de les afficher/utiliser, selon le contexte de destination.

// Contexte HTML — OWASP Java Encoder
String html = Encode.forHtml(userInput);
// < → &lt;   > → &gt;   " → &quot;
// <script> → &lt;script&gt; (inoffensif)

// Contexte SQL → PreparedStatement (jamais concat)
// Contexte JavaScript
String js = Encode.forJavaScript(userInput);
// Contexte URL
String url = Encode.forUriComponent(userInput);
  • L'échappement dépend du contexte de sortie
  • HTML ≠ JavaScript ≠ SQL ≠ Shell ≠ URL
  • Utiliser les fonctions des frameworks (pas du code maison)

Contextes d'échappement — Tableau récapitulatif

Contexte Caractères dangereux Transformation Outil recommandé
HTML < > " ' & &lt; &gt; &quot; &#x27; &amp; OWASP Encoder / Thymeleaf auto
SQL ' " ; -- Paramètre lié (pas concaténé) PreparedStatement / ORM
JavaScript </script> \ ' " Séquences \uXXXX Encode.forJavaScript()
URL espaces & = ? # Encodage %XX URLEncoder / encodeURIComponent
Shell ; | & ` $ ( ) Éviter tout appel shell avec input user ProcessBuilder (sans shell)

❌ Anti-pattern courant — XSS Stocké

// Commentaire utilisateur stocké sans sanitisation :
// input = <img src=x onerror="steal(document.cookie)">
// → Stocké tel quel en BD
// → Affiché tel quel dans la page HTML
// → Exécuté dans le navigateur de CHAQUE visiteur !

→ Vol de session de tous les utilisateurs

✅ Pattern correct

// 1. Sanitiser en entrée (supprimer balises dangereuses)
String clean = sanitizer.sanitize(input);
// 2. Stocker la version nettoyée
commentRepo.save(new Comment(clean));
// 3. Échapper en sortie (Thymeleaf le fait auto)
// <p th:text="${comment.text}"></p>
// th:text échappe automatiquement → &lt;img...&gt;
// Affiché comme texte, jamais exécuté
⚠️ Piège : Sanitiser à l'entrée ne suffit pas — si vous concaténez dans du SQL ou du HTML sans échapper, vous restez vulnérable. Les deux étapes sont indépendantes et toutes deux nécessaires.

❌ Code JavaScript exécuté chez tous les visiteurs

🔍 Pourquoi ça marche ?

Sans échappement, le serveur génère ce HTML :

<!-- Page HTML générée — DANGEREUX -->
<p>Message de Kevin :</p>
<p>
  <script>alert('Vous êtes piraté !')</script>
</p>
<!-- Le navigateur voit du <script>
     et l'exécute immédiatement ! -->

✅ Avec échappement — Inoffensif

Le serveur transforme les < et > avant d'afficher :

<!-- Page HTML générée — SÉCURISÉ -->
<p>Message de Kevin :</p>
<p>
  &lt;script&gt;alert('...')&lt;/script&gt;
</p>
<!-- Le navigateur affiche le texte
     tel quel — rien n'est exécuté ✅ -->
📝
L'attaquant
tape du code HTML/JS dans un champ texte ordinaire
🔐
L'échappement
transforme < en &lt; — le navigateur lit du texte, pas du code
✅
Résultat
Le message s'affiche comme du texte inoffensif — jamais exécuté

Rendu Serveur vs Rendu Client — Impact Sécurité

Pourquoi c'est important : Le lieu où la page est construite détermine où l'échappement doit avoir lieu — et qui est responsable de la protection contre le XSS.

🖥️ Rendu Côté Serveur (SSR)

Le serveur assemble le HTML complet, puis l'envoie au navigateur.

👤
Navigateur
fait une requête
→
⚙️
Serveur
génère le HTML
→
📄
HTML complet
reçu et affiché
<!-- Thymeleaf (Java) — échappement auto -->
<p th:text="${user.comment}"></p>
<!-- th:text échappe < > " automatiquement -->

<!-- th:utext = PAS d'échappement → dangereux -->
<p th:utext="${user.comment}"></p>
  • Exemples : Thymeleaf Django PHP Rails JSP
  • L'échappement se fait sur le serveur, avant l'envoi
  • Le navigateur reçoit du HTML déjà sécurisé

🌐 Rendu Côté Client (CSR)

Le navigateur reçoit du JavaScript, qui construit la page dynamiquement.

👤
Navigateur
reçoit JS + JSON
→
⚛️
React/Vue/Angular
construit le DOM
→
🖼️
Page rendue
dans le navigateur
// React — échappement automatique via JSX ✅
<p>{user.comment}</p>
// JSX échappe les < > automatiquement ✅

// dangerouslySetInnerHTML → PAS d'échappement ❌
<p dangerouslySetInnerHTML=
  {{__html: user.comment}} />
// Le nom dit tout : "dangerously" ⚠️
  • Exemples : React Vue Angular Svelte
  • L'échappement se fait dans le navigateur, par le framework JS
  • Les données viennent d'une API REST/GraphQL → valider côté serveur aussi
Critère SSR — Côté Serveur CSR — Côté Client
Où est généré le HTML ? Sur le serveur (Java, Python, PHP…) Dans le navigateur (JavaScript)
Où échapper ? Côté serveur, avant l'envoi Côté client, via le framework JS
Risque si mal fait XSS Réfléchi / Stocké DOM-based XSS
API exposée ? Non (HTML final direct) Oui — API REST/GraphQL à sécuriser aussi
Frameworks qui échappent auto Thymeleaf th:text, Django {{ }}, Rails ERB React JSX {}, Angular {{ }}, Vue {{ }}

A03 — Injection SQL : Comprendre & Corriger

Code Vulnérable

// DANGEREUX — concaténation directe
String q = "SELECT * FROM users WHERE " +
  "login='" + login + "' AND " +
  "password='" + password + "'";

// Attaque : login = "admin' --"
// → SELECT * FROM users WHERE
//   login='admin' --' AND password='x'
// L'attaquant contourne l'authentification !

Code Sécurisé — Requêtes paramétrées

// SÉCURISÉ — PreparedStatement
PreparedStatement ps = conn.prepareStatement(
  "SELECT * FROM users " +
  "WHERE login = ? AND password = ?"
);
ps.setString(1, login);
ps.setString(2, bcrypt(password));
ResultSet rs = ps.executeQuery();
// Les ? ne sont jamais exécutés comme SQL
Prévention : Requêtes paramétrées ORM (Hibernate, JPA) Validation des entrées Moindre privilège BDD WAF SAST (SonarQube)

XSS & CSRF — Attaques côté Client

XSS — Cross-Site Scripting

Réfléchi (Reflected) — Script dans URL, exécuté immédiatement. Vecteur de phishing.
Stocké (Stored) — Script en BDD, exécuté à chaque visite. Le plus dangereux.
DOM-based — Manipulation côté client via JS vulnérable.

Prévention XSS

  • Échapper les sorties HTML
  • Content Security Policy (CSP)
  • HttpOnly & Secure cookies
  • DOMPurify pour le HTML dynamique

CSRF — Cross-Site Request Forgery

Principe : Forcer un utilisateur authentifié à exécuter une action non désirée sur le site cible (virement, changement de mot de passe…)
Exemple d'attaque : 
<img src="https://bank.com/transfer
  ?to=attacker&amount=9999">

Prévention CSRF

  • Tokens CSRF (synchronizer token pattern)
  • SameSite cookie attribute
  • Vérifier l'en-tête Origin/Referer
  • Double Submit Cookie

Pratiques de Codage Sécurisé

Validation · Chiffrement · Secrets · Revue de code

Principes de Codage Sécurisé

① Validation des Entrées

Valider TOUT input (forms, APIs, URL params, headers, cookies). Utiliser des listes blanches (whitelist). Valider type, taille, format, plage de valeurs.

② Encodage des Sorties

Encoder les données selon le contexte de sortie (HTML, URL, JavaScript, SQL). Utiliser des fonctions dédiées plutôt que du code manuel.

③ Authentification & Sessions

MFA obligatoire · Hachage (bcrypt, Argon2) · Tokens sécurisés · Expiration · Invalidation à la déconnexion

④ Gestion des Erreurs

Ne jamais exposer les détails techniques en production. Logger côté serveur, afficher des messages génériques côté client. Toujours fail securely.

⑤ Chiffrement des Données

TLS 1.3 en transit · AES-256-GCM au repos · Jamais de mots de passe en clair · Rotation régulière des clés

⑥ Gestion des Dépendances

Scanner régulièrement (SCA) · Lockfile (versions fixes) · Supprimer les dépendances inutilisées · SBOM obligatoire

Gestion des Secrets — Ne Jamais Coder en Dur !

INTERDIT — Hardcoded secrets

// NE JAMAIS FAIRE ÇA !
String apiKey = "sk-1234567890abcdef";
String dbPass = "MySecretPass123!";
String jwtSecret = "my-secret-key";

// Ces secrets finissent dans Git !
// Indexés par GitHub/GitLab
// Exposés pour toujours dans l'historique

CORRECT — Variables env / Vault

// Variables d'environnement
String apiKey = System.getenv("API_KEY");

// Spring Boot — application.yml
@Value("${db.password}")
private String dbPassword;

// HashiCorp Vault
VaultResponse r = vault.read("secret/app");
String secret = (String) r.getData()
                           .get("password");
Bonnes pratiques : .gitignore pour .env GitLeaks en pre-commit Rotation automatique Vault / AWS Secrets Manager Moindre accès aux secrets Audit trail accès secrets

Stratégie de Tests de Sécurité

SAST — Boîte Blanche

Analyse code source sans exécution · Détecte au niveau du code
Quand : CI/CD au commit (shift-left)
Outils : SonarQube, Semgrep, Checkmarx, CodeQL
Trouve : SQL injection, XSS, secrets hardcodés, null pointers

DAST — Boîte Noire

Teste l'app en exécution · Simule un attaquant externe
Quand : Environnement de staging, avant release
Outils : OWASP ZAP, Burp Suite, Nikto, Nuclei
Trouve : Injections runtime, auth bypass, misconfig

SCA — Analyse des Composants

Analyse dépendances tierces pour CVE connues
Quand : Continu (à chaque mise à jour de deps)
Outils : Snyk, OWASP Dep-Check, Trivy
Trouve : CVE dans librairies, licences problématiques

Pentest — Boîte Grise/Noire

Test manuel par un expert sécurité
Quand : Annuel ou avant mise en production majeure
Outils : Metasploit, Burp Pro, sqlmap, Nmap
Trouve : Vulnérabilités logiques, chaînes d'exploitation
Approche complémentaire : SAST (shift-left) + DAST (staging) + SCA (continu) + Pentest (annuel) = couverture maximale

JWT — JSON Web Token

Structure : Header.Payload.Signature
Stateless · Signé (HS256/RS256/ES256) · Expirant
Risques : alg:none, faibles clés, pas de rotation, révocation impossible
Best practice : RS256, exp court, refresh token rotation

Authorization Code Flow + PKCE · Scopes minimaux
Refresh token rotation · Token introspection
Implémentations : Keycloak, Auth0, Okta, Spring Authorization Server

API Gateway

Rate limiting · Auth centralisée · WAF intégré · Logging
Outils : Kong, AWS API Gateway, Nginx, Traefik

Zero Trust & Sécurité Cloud

"Never Trust, Always Verify"

Pas de périmètre de confiance implicite. Chaque requête est authentifiée, autorisée et vérifiée — peu importe l'origine (interne ou externe).

Principes Zero Trust

  • Vérifier explicitement chaque identité
  • Moindre privilège d'accès à tout moment
  • Supposer la compromission (assume breach)
  • Microsegmentation réseau
  • MFA partout, y compris infra interne
  • Surveillance et logging continus
